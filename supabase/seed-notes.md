# Seed Notes — Clearview Square Supabase Migration

## Source of Truth

The JSON files in `public/data/` remain the authoritative source for seed data:

- `public/data/stores.json` — 30 store records (29 public, 1 hidden)
- `public/data/promotions.json` — 4 sample promotion/event records

Run `npm run validate:data` before any seed operation. Do not seed from data that fails validation.

## When to Seed

Seed only after:
1. Supabase project has been provisioned.
2. `supabase/schema.sql` has been applied (tables exist).
3. `supabase/rls-policies.sql` has been applied (RLS is active).
4. `npm run validate:data` passes cleanly.

## Seed Script (Not Yet Created)

The seed script (`scripts/seed-supabase.js`) has not been written yet. It will be created in a later phase when explicitly approved.

When created, the script will:
- Read `public/data/stores.json` and `public/data/promotions.json`
- Transform JSON to database rows (see transformations below)
- Insert records using the Supabase service role client
- Log success/failure per record

The script will be Node.js ESM, matching the existing `scripts/` convention.

### Trading Hours

The JSON `tradingHours` field is a flat object:

```json
{
  "monday": "08:00 - 20:00",
  "tuesday": "08:00 - 20:00",
  "saturday": "08:00 - 17:00",
  "sunday": "Closed",
  "publicHolidays": "08:00 - 15:00"
}
```

Each key becomes one row in `trading_hours`. The `sort_order` maps to day position:

| day_key         | display_label    | sort_order |
|-----------------|------------------|------------|
| `monday`        | Monday           | 0          |
| `tuesday`       | Tuesday          | 1          |
| `wednesday`     | Wednesday        | 2          |
| `thursday`      | Thursday         | 3          |
| `friday`        | Friday           | 4          |
| `saturday`      | Saturday         | 5          |
| `sunday`        | Sunday           | 6          |
| `publicHolidays`| Public Holidays  | 7          |

The freeform value (`"08:00 - 20:00"`, `"Closed"`, `"24 Hours"`, `"Opening Soon"`, `"N/A"`) goes into the `notes` column.

`open_time` and `close_time` are **left null** during Phase 4 seed. These will be populated later when structured time data is confirmed with centre management.

### store_id Resolution

JSON promotions use string `storeId` (e.g. `"checkers"`). The DB `store_id` is a UUID. The seed script must:
1. Insert all stores first.
2. Build a lookup map of `slug → uuid` from the inserted rows.
3. Resolve each promotion's `storeId` against the slug lookup to get the UUID.

### Known Issue: storeName Denormalisation

The JSON `promotions.storeName` field is not stored in Supabase. The admin and public UI must derive store name via `store_id → stores.name` join. The `Promotions.jsx` page already has a `getStoreName(storeId)` function that will need updating in Phase 4B to use the joined store name from Supabase.

## Credentials Required for Seeding

The seed script requires the **Supabase service role key** to bypass RLS and insert seed data.

```env
# NOT VITE_ prefixed — this is a Node.js-only variable, never bundled into the frontend
SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>
```

**Rules:**
- Store this key in your local `.env` file only.
- `.env` is gitignored — confirm before adding the key.
- Never use `VITE_SUPABASE_SERVICE_ROLE_KEY` — the `VITE_` prefix causes Vite to bundle the value into the client-side JavaScript, which would expose it publicly.
- Never commit this key. Never paste it into any repo file.
- The Supabase dashboard → Project Settings → API → Service role key (secret).

## Post-Seed Verification

After seeding, run a lightweight check:
- Query `stores` with the anon key via Supabase JS client → should return only `published` and `opening_soon`, `is_visible = true` records (29 public records, not 30).
- Query `promotions` with anon key → should return only `published` records (4 in current data).
- Query `trading_hours` → should return 8 rows per store (30 stores × 8 days = 240 rows).
- Confirm hidden store (`milky-lane-storage`) does not appear in anon-key queries.

## Rollback

The JSON files are never modified by the seed process. If Supabase proves unsuitable, the public pages revert to JSON fetching by reverting the hooks introduced in Phase 4B. No data is lost.
